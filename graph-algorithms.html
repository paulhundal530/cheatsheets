01 — Memory

Three ways to represent a graph

The representation you choose dictates the cost of every later operation. For V vertices and E edges, adjacency lists win for the sparse graphs (E ≪ V²) you'll see in almost every interview.

Adjacency List

The default. Array/map of neighbor lists.

O(V+E) space
  • Iterate a node's neighbors in O(deg)
  • Space-optimal for sparse graphs
  • Fast to build; the interview go-to
  • Edge lookup (u,v)? is O(deg), not O(1)

Adjacency Matrix

V×V grid; cell = edge weight / bool.

O(V²) space O(1) edge lookup
  • Check/insert/delete an edge in O(1)
  • Great for dense graphs & math (powers, Floyd-Warshall)
  • Wastes memory on sparse graphs
  • Listing neighbors is O(V) — scans a whole row

Objects & Pointers

Node objects referencing neighbor objects.

O(V+E) space
  • Natural in OOP; mirrors real domain models
  • Node carries its own payload & methods
  • No global index — hard to iterate all nodes
  • Pointer-chasing hurts cache locality
Rule of thumb  Sparse graph → adjacency list. Dense graph or constant-time edge queries → adjacency matrix. Modeling a domain (game map, org chart) → objects & pointers.

02 — Traversal

Breadth-first & depth-first search

Both visit every vertex and edge once — O(V+E) time on an adjacency list, O(V²) on a matrix. The difference is the frontier data structure: a queue spreads out level by level, a stack (or recursion) plunges deep.

BFS — Breadth-First

Queue · explores nearest first · shortest path on unweighted graphs

O(V+E) time O(V) space
  • Best for shortest path in # of edges, level-order, "nearest" queries
  • Mark visited on enqueue, not dequeue — else nodes get queued twice

DFS — Depth-First

Stack / recursion · explores deep first · backtracking

O(V+E) time O(h) space
  • Best for cycle detection, topological sort, connected components, path existence
  • Recursion depth = longest path; deep graphs can overflow the call stack → go iterative

Iterative DFS

An explicit stack avoids call-stack overflow on large or adversarial graphs.

04 — Weighted shortest path

Dijkstra's algorithm

Shortest path from a source on a graph with non-negative edge weights. It's BFS with a priority queue: always expand the closest unfinalized node. Once a node is popped, its distance is final (the greedy invariant that negative edges would break).

Lazy Dijkstra with a binary heap

O((V+E) log V) time O(V) space
  • Requires all edge weights ≥ 0. Negative edges → use Bellman-Ford O(V·E).
  • "Lazy" deletion: push duplicates, skip a popped entry when its distance is stale.
Whiteboard note  With a Fibonacci heap it's O(E + V log V), but binary-heap Dijkstra is what you write in an interview.

05 — Informed search

A* search

Dijkstra guided toward a goal by a heuristic. It orders the frontier by f(n) = g(n) + h(n) — cost so far plus an estimate of cost remaining — so it explores toward the target instead of in all directions.

A* on a weighted graph

O(E log V) typical O(b^d) worst case
  • Admissible h never overestimates → A* returns the optimal path.
  • Consistent h(u) ≤ w(u,v) + h(v) → no node needs re-expansion.
  • h(n) = 0 collapses A* into Dijkstra; a perfect h walks straight to the goal.
Where it wins  Grid/road pathfinding where you can cheaply estimate distance-to-goal — e.g. Manhattan or Euclidean distance as h.

08 — Graphs in disguise

Grid & matrix problems

A 2D grid is a graph: each cell is a vertex, and edges connect it to its neighbors. You rarely build an explicit graph — you traverse the grid directly, using a direction vector for the neighbors. V = rows·cols, and each cell has ≤ 4 (or 8) edges, so traversal is O(rows·cols).

The key trick  Encode neighbors as a direction array instead of writing four if blocks. Use 4-directional for orthogonal moves, 8-directional when diagonals count.

Count Islands — DFS flood fill

Connected components on a grid · sink each island as you find it

O(R·C) time O(R·C) space
  • Every unvisited land cell starts a new component; flood it to mark visited
  • Mutating the grid to '0' avoids a separate visited array

Shortest Path in a Grid — BFS

Fewest steps through open cells · unweighted ⇒ BFS is optimal

O(R·C) time O(R·C) space
  • BFS layer count = shortest number of steps (weights are all 1)
  • Mark visited on enqueue; the first time you pop the target, that's the answer
Escalate when…  Cells have costs (not all 1) → swap BFS for Dijkstra. Costs are only 0 or 1 (e.g. "walls you may break") → 0-1 BFS with a deque: push cost-0 moves to the front, cost-1 to the back for O(R·C). Need multi-source spread (rotting oranges, nearest exit) → seed the queue with all sources before starting BFS.

09 — Worth knowing

Two building blocks that show up everywhere

Topological sort and Union-Find are the backbone of dependency, ordering, and connectivity problems.

Topological Sort — Kahn's

Linear order of a DAG so every edge points forward. Empty result ⇒ a cycle exists.

O(V+E) time

Union-Find (DSU)

Near-constant connectivity queries with path compression + union by rank.

O(α(V)) per op
