01 — Language

Kotlin language essentials

The keyword-level questions. Know the why, not just the definition — that's what separates a rehearsed answer from an understood one.

QWhat is an inline function?

The compiler copies the function's body — and the bodies of its lambda arguments — directly into each call site instead of generating a function object per lambda.

  • Removes the per-lambda allocation and virtual-call overhead of higher-order functions
  • Enables non-local returns from a lambda and is required for reified type params
  • Bloats bytecode if the function is large or called in many places — reserve it for small higher-order functions

Modifiers: noinline excludes one lambda param; crossinline keeps the lambda inlined but forbids non-local returns.

QWhat is the reified keyword?

Inside an inline function, reified T keeps the type argument available at runtime, defeating JVM type erasure. You can then use T::class, is T, and as T without passing a Class<T>.

inline fun <reified T> Gson.fromJson(json: String): T =
    fromJson(json, T::class.java)

val user: User = gson.fromJson(body)  // no User::class arg

Catch: only works because the body is inlined — reified requires inline.

QAdvantage of const?

const val is a compile-time constant: its value is inlined into the bytecode at every use site, with no backing field or getter.

  • Faster (no accessor call) and usable in annotations and when labels, where a plain val can't go
  • Restricted to top-level or object/companion members, and only primitives or String

vs val: a plain val is assigned at runtime and read through a generated getter.

Qlateinit vs lazy?

Accessing lateinit before assignment throws UninitializedPropertyAccessException; guard with ::field.isInitialized.

Q== vs ===?

  • == — structural equality; calls equals() (with a null check). "Do these hold the same value?"
  • === — referential equality; same object identity on the heap. "Are these the exact same instance?"

Java's == is Kotlin's ===; Java's .equals() is Kotlin's ==.

QWhat does open do?

Kotlin classes and members are final by default. open opts a class in to being subclassed, or a member in to being overridden.

It's the inverse of Java, where everything is open unless marked final. The default nudges you toward composition and stable APIs.

QThe internal visibility modifier?

Visible everywhere within the same module — one compilation unit (a Gradle module, Maven project, IntelliJ module).

Broader than private/protected, narrower than public. Perfect for library code you share across your own module but don't want to leak as public API.

QThe init block?

Initializer code that runs as part of the primary constructor. Multiple init blocks and property initializers execute top-to-bottom in declaration order.

class User(name: String) {
    val slug: String
    init {
        require(name.isNotBlank())   // validate during construction
        slug = name.lowercase().trim()
    }
}

02 — Interop

Kotlin ↔ JVM interop annotations

These exist so Java callers see idiomatic Java. If your code is only ever called from Kotlin, you rarely need them — that framing is the answer interviewers want.

Q@JvmStatic

On a companion object or object member, emits a genuine static method/field on the enclosing class.

Java can call Foo.bar() instead of Foo.Companion.bar().

Q@JvmField

Exposes a property as a plain public field — no getter/setter generated.

For interop (and micro-optimizations). Incompatible with open, override, const, and private.

Q@JvmOverloads

Java doesn't understand default arguments. This generates one overload per default parameter so Java can omit them.

@JvmOverloads
fun greet(name: String, loud: Boolean = false) {}
// Java sees: greet(name) and greet(name, loud)

03 — Concurrency

Kotlin coroutines

Coroutines are about not blocking threads. Every question here comes back to that idea.

QSuspending vs blocking?

  • Blocking — the thread is parked and can do nothing else until the call returns. One thread held hostage.
  • Suspending — a suspend function can pause its coroutine and release the thread back to the pool, resuming later on any thread.

So thousands of suspended coroutines can share a handful of threads; thousands of blocking calls need thousands of threads.

Say it  Suspension is cooperative and cheap; blocking wastes a whole thread. delay() suspends, Thread.sleep() blocks.

Qlaunch vs async?

  • launch → returns a Job. Fire-and-forget, no result. For side effects.
  • async → returns a Deferred<T>. A promise of a value you await(). For concurrent work that produces results.
val user  = async { api.user() }
val posts = async { api.posts() }
render(user.await(), posts.await())  // run in parallel

Exceptions: launch throws into the parent immediately; async holds the exception until you await().

QCommon concurrency bugs — and how to diagnose them

Frequent culprits

  • Main-thread blocking — network/DB/JSON on Dispatchers.Main → ANR & jank
  • Races on shared mutable state — two coroutines mutating the same var
  • Leaked coroutines from GlobalScope or no structured scope — they outlive the screen
  • Ignored cancellation — a CPU loop that never checks isActive
  • Swallowed exceptions — an async result that's never await()ed

How to diagnose

  • StrictMode catches disk/network on the main thread the instant it happens
  • Log Thread.currentThread().name; enable coroutine debug mode for named stacktraces
  • CPU Profiler thread timeline; read ANR traces (Play Console or /data/anr)
  • LeakCanary for leaked jobs; runTest + a test dispatcher for deterministic ordering

Fixes Right dispatcher (IO for blocking, Default for CPU); confine mutable state and guard it with a Mutex or atomics; prefer immutable data + StateFlow; lean on structured concurrency so cancellation propagates automatically.

class Counter {
    private val mutex = Mutex()
    private var count = 0
    suspend fun inc() = mutex.withLock { count++ }   // no races
}

04 — Framework

Components & lifecycle

Where "I memorized the diagram" meets "I understand why the callback fires." The gotchas below are the ones that get asked.

QWhat is a ViewModel and why use it?

A lifecycle-aware holder for UI state and logic that survives configuration changes (rotation, locale) — it isn't recreated when its Activity/Fragment is.

  • Separates UI from business logic (the VM in MVVM); pairs with StateFlow/LiveData
  • Scoped to a lifecycle owner; cleared in onCleared() when that owner is gone for good
  • Never hold a View or Activity Context — that leaks. Use AndroidViewModel for app context.

QWhy call setContentView() in onCreate()?

onCreate() is the one-time init callback — it runs once per Activity instance, before the activity is visible. That's the moment to inflate the layout and attach the view tree to the window.

Putting it in onStart()/onResume() would re-inflate the whole hierarchy on every return to the foreground — wasteful and state-destroying.

QWhy use only the default constructor for a Fragment?

The system recreates fragments by reflection using the no-arg constructor (after a config change or process death). A custom constructor is never called on recreation, so its arguments are lost — often crashing with an InstantiationException.

Do this instead  Pass data via a Bundle in a newInstance() factory + setArguments(); it's restored automatically on recreation.

QWhen is only onDestroy() called — no onPause()/onStop()?

When you call finish() inside onCreate() (or before the activity ever becomes visible). It never reaches the started/resumed state, so the system goes straight onCreate() → onDestroy(), skipping pause and stop.

(A system process-kill can also destroy an activity without guaranteed callbacks — but the deterministic case is finish() in onCreate().)

QsingleTask launch mode?

The activity has at most one instance, living at the root of its task. A new intent for it reuses that instance — routing through onNewIntent() — and clears any activities above it.

The four modes: standard (new instance each time), singleTop (reuse only if already on top), singleTask (one per task, clear-to-top), singleInstance (alone in its own task).

QThe Activity lifecycle, in order?

onCreate → onStart → onResume (now interactive) → onPause → onStop → onDestroy. Returning from stopped fires onRestart → onStart.

  • onStart/onStop bracket visibility; onResume/onPause bracket foreground focus
  • Persist state in onSaveInstanceState / release resources in onStop

QHow is a ViewModel lifecycle-aware?

It doesn't listen to lifecycle callbacks itself — it's scoped to a lifecycle owner and knows the difference between a temporary teardown and a permanent one.

  • An Activity/Fragment is a ViewModelStoreOwner holding a ViewModelStore — a map of its ViewModels.
  • On a configuration change the framework retains that store and hands it to the new Activity instance, so you get the same ViewModel back rather than a fresh one.
  • onCleared() runs only when the owner is finishing for good (Activity isFinishing, Fragment permanently removed) — never on rotation.
  • viewModelScope is a CoroutineScope cancelled in onCleared(), so its coroutines stop at exactly the right moment.
Limit  It survives rotation but not process death — for that, persist through a SavedStateHandle.

QHow does Android handle device rotation?

Rotation is a configuration change. By default the system destroys and recreates the Activity so it can load orientation-specific resources (layout-land/, values-land/): onPause → onStop → onSaveInstanceState → onDestroy → onCreate → … → onRestoreInstanceState.

Three layers preserve your state across that recreation

  • View state — views with an id auto-save scroll position, text, and selection
  • onSaveInstanceState(Bundle) — small, transient UI state; survives rotation and process death (keep it small & Parcelable)
  • ViewModel — in-memory state that's costly to rebuild; survives rotation (add SavedStateHandle to also survive process death)
override fun onSaveInstanceState(out: Bundle) {
    super.onSaveInstanceState(out)
    out.putInt("scrollY", scrollY)   // restored in onCreate / onRestoreInstanceState
}

Opt out Declaring android:configChanges="orientation|screenSize" makes the system call onConfigurationChanged() instead of recreating — use sparingly, since you then own every resource swap. In Compose, rememberSaveable restores state across rotation.

05 — Platform

System internals

How the platform actually delivers messages, packs your code, and manages memory.

QHow does the Android push-notification system work (FCM)?

Every app's notifications ride a single persistent socket that Google Play Services keeps open to Google's servers — multiplexing all apps over one connection to save battery. The flow:

StepWhat happens
1 · RegisterApp requests an FCM registration token (FirebaseMessaging.getInstance().token) — a unique address for this app install.
2 · StoreApp sends that token to your backend and saves it. Handle rotation in onNewToken().
3 · SendYour server calls the FCM HTTP v1 API, targeting a token, a topic, or a device group.
4 · DeliverFCM pushes down the persistent connection to the device (respecting Doze / priority).
5 · HandleYour FirebaseMessagingService.onMessageReceived() runs; you build a Notification on a NotificationChannel.
 
class MyFcmService : FirebaseMessagingService() {
    override fun onNewToken(token: String) {
        sendTokenToServer(token)
    }
    override fun onMessageReceived(msg: RemoteMessage) {
        val n = NotificationCompat.Builder(this, CHANNEL_ID)
            .setContentTitle(msg.data["title"])
            .setContentText(msg.data["body"])
            .setSmallIcon(R.drawable.ic_note)
            .build()
        NotificationManagerCompat.from(this).notify(1, n)
    }
}
Two message types  Notification messages are auto-shown by the system tray when the app is backgrounded; data messages always land in onMessageReceived() for you to handle. Send data-only if you want full control.

QWhat is Multidex?

A single DEX file can reference at most 65,536 methods (the "64K limit"). Multidex splits your app across multiple DEX files to get past it.

  • API 21+ (ART): multidex is native — handled at build time, nothing to do
  • API < 21: add the multidex library and use MultiDexApplication

Better fix: shrink method count with R8/ProGuard and fewer dependencies before reaching for it.

QCan you force garbage collection?

You can only request it — System.gc() / Runtime.getRuntime().gc() are hints the VM is free to ignore. There's no way to force a deterministic collection.

  • Discouraged in production: it can trigger a stop-the-world pause and hurts performance

The real answer is to fix leaks and reduce allocations (LeakCanary, the Memory Profiler) rather than nudge the GC.

QHow does Kotlin Multiplatform (KMP) work?

Write your business logic once — networking, data, domain, use-cases — and share it across Android, iOS, desktop, web, and backend, while keeping each platform's native UI.

  • commonMain holds shared code; platform gaps are bridged with expect (declaration) / actual (per-platform implementation)
  • Compiles to JVM bytecode for Android, native binaries for iOS via Kotlin/Native, and JS/Wasm for web
  • Compose Multiplatform can additionally share the UI layer when you want to

Why it wins: one source of truth for logic (fewer bugs, less duplication) without giving up native performance or platform look-and-feel.

07 — Performance

Diagnosing UI jank & performance

Smooth means hitting the frame budget: about 16 ms per frame at 60 fps (~8 ms at 120 fps). Miss it and the user sees stutter, lag, or flicker — and the cause is almost always too much work on the main thread.

QWhat makes the UI jank?

  • Blocking work on the main thread — I/O, DB queries, JSON parsing, bitmap decoding, big loops
  • Deep view hierarchies & overdraw (stacked opaque backgrounds) → slow measure / layout / draw
  • Heavy onBindViewHolder, or notifyDataSetChanged() rebinding the whole list
  • Allocations inside onDraw / onBindViewHolder → GC pauses mid-scroll
  • Large, un-downsampled bitmaps; layout thrashing from repeated requestLayout()

QFixing scroll stutter & flicker

  • Do all data prep off the main thread; bind pre-computed view state
  • Use DiffUtil / ListAdapter instead of notifyDataSetChanged() — updates only changed rows
  • setHasStableIds(true) so recycled rows keep their identity — this kills most flicker
  • Keep onBindViewHolder allocation-free; no inflation or I/O in there
  • Image lib (Coil/Glide) with downsampling, caching, and cancel-on-recycle — stops the wrong image flashing in
  • Big lists → Paging 3; add setHasFixedSize(true) and a shared RecycledViewPool

08 — Rounding out

More worth knowing

High-frequency follow-ups that pair naturally with the questions above.

Qdata class — what do you get?

The compiler generates equals()/hashCode() (value-based), toString(), copy(), and componentN() for destructuring — all from the primary-constructor properties.

data class User(val id: Int, val name: String)
val u2 = u.copy(name = "Ada")
val (id, name) = u2         // destructuring

Qsealed class/interface vs enum?

A sealed type has a fixed, compiler-known set of subtypes in the same module — but each subtype is a full class that can hold its own data and multiple instances.

  • Makes when exhaustive with no else — add a case and the compiler flags every when
  • Ideal for UI state: Loading / Success(data) / Error(msg)

QScope functions: let / run / with / apply / also?

apply for configuring an object; let for null-safe transforms (x?.let { }).

QStateFlow vs LiveData?

  • LiveData — Android-only, lifecycle-aware out of the box, always has value on the main thread
  • StateFlow — pure Kotlin (KMP-friendly), always has a value, richer operators; collect with repeatOnLifecycle to stay lifecycle-safe

Modern code favors StateFlow/SharedFlow; LiveData remains common in older codebases.

QKotlin null safety?

Types are non-null by default; String? opts in to null. The toolkit:

  • ?. safe call · ?: Elvis default · ?.let{} run-if-non-null
  • !! assert-non-null (throws NPE) — a code smell to justify

QWhat is companion object?

A single object tied to a class — Kotlin's stand-in for static. Holds factory methods and constants, can implement interfaces, and is initialized lazily on first class access.

Members need @JvmStatic/@JvmField to look static from Java.

QStructured concurrency & scopes?

Coroutines launch into a CoroutineScope; the scope owns them, so cancelling it cancels all children and it waits for them to finish.

  • viewModelScope — cancelled in onCleared()
  • lifecycleScope — tied to a UI lifecycle owner
  • Dispatchers: Main (UI), IO (network/disk), Default (CPU)

QCommon causes of memory leaks?

  • A ViewModel/singleton/static holding a View or Activity Context
  • Non-static inner classes, Handlers, or listeners outliving their Activity
  • Coroutines/RxJava subscriptions not cancelled with the lifecycle

Detect: LeakCanary + the Memory Profiler. Prevent: use lifecycle-scoped work and app Context where possible.
