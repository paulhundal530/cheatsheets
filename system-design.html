System Design Patterns · interview cheat sheet

Eight patterns cover almost every system design interview.

Identify the pattern first, then walk its solution ladder from simple to complex — naming the technology and the trade-off at each rung. The four fundamentals every answer rests on

Before you reach for a pattern, frame the problem along these four axes — interviewers listen for them by name. Each card ends with the sentence worth saying out loud.

Storage & movement

Data Flow

  • Trace one request end to end — client → load balancer → API → cache → database → back. Name every hop and where the data rests.
  • SQL for relations, transactions, and ad-hoc queries; NoSQL (key-value, document, wide-column) for scale, flexible schema, or a single known access pattern.
  • Index the columns you filter and sort on; denormalize to kill joins on the hot read path.
  • Keep the hot path in memory — low latency is a data-placement problem, not a faster-CPU one.

Say: reads and writes have different shapes — model each for its own access pattern.

Throughput

Scalability

  • Vertical (bigger box) is simplest but caps at one machine; horizontal (more boxes) scales without limit and adds coordination.
  • Keep services stateless so they scale out behind a load balancer; push state down to the datastore.
  • Cache reads (Redis, CDN); shard and partition writes by a key that spreads load evenly.
  • The four levers: load balancing, read replicas, caching, and database partitioning.

Say: scale the tier that's actually the bottleneck — measure before you scale.

Reliability

Fault Tolerance

  • No single point of failure: redundant instances across zones, health checks, and automatic failover.
  • Replicate data (leader + followers); a follower is promoted the moment the primary is lost.
  • Contain the blast radius with timeouts, retries, circuit breakers, and bulkheads.
  • Degrade gracefully — serve stale cache or a reduced experience instead of a hard error.

Say: assume every component will fail — redundancy plus failover keeps the service up.

The trade-off

Consistency vs Availability

  • A partition will happen (CAP) — you only choose what breaks. CP: stall or reject to stay correct (payments, inventory, bookings). AP: keep serving and reconcile later (feeds, likes, presence).
  • Strong consistency = read-your-writes and higher latency; eventual = fast, cheap, briefly stale.
  • Choose per feature, not per system — a bank balance is CP, the like counter beside it can be AP.
  • Replication lag is this choice in disguise: write to primary, read a lagging replica, get stale data.

Say: name which side the feature needs and why — that is the entire question.

The patterns

Scaling Reads▶ Read traffic dwarfs writes — serve the same data to millions without melting the database.

Spot it when

  • One Instagram feed load fires 100+ queries while the user writes once a day.
  • The interviewer says "millions of daily active users" but writes are rare.
  • The same popular content is requested over and over (hot keys).

Solution ladder — climb only as far as needed

  1. Optimize inside the database. Add indexes on query columns, tune slow queries, denormalize to avoid joins, upgrade hardware. Cheapest wins first.
  2. Add read replicas. Copies of the database serve reads; the primary handles writes. Scales horizontally but introduces replication lag.
  3. Add external caching. Redis in front of the database for hot data; a CDN at the edge for static or cacheable content. Accept some staleness in exchange for massive throughput.

Say the order out loud — jumping straight to "add a cache" without trying indexes first reads as unseasoned.

Tech to name & why

Postgres/MySQLStart here; indexes and denormalization go surprisingly far.
Read replicasNative to most RDBMSs; spread read load with no app rewrite.
RedisIn-memory cache for hot keys; sub-millisecond reads.
CDNPushes cacheable content to the edge, off your origin entirely.
ElasticsearchOffload text-search reads the primary database is bad at.

Deep dives to expect

  • Cache invalidation — how fresh must data be? TTLs vs. write-through vs. explicit invalidation.
  • Replication lag — user writes then immediately reads a stale replica; read-your-own-writes.
  • Cache stampede — a hot key expires and thousands of requests rebuild it at once; use locks or early refresh.
  • Hot keys — one celebrity post overwhelms a single cache node.

Instagram / news feedsProduct catalogsURL shortener (Bitly)Twitter timelines
Scaling Writes▶ Write volume exceeds what one database can absorb — partition, buffer, and batch.

Spot it when

  • High-throughput ingestion: clickstreams, metrics, GPS pings, likes.
  • Bursty traffic — flash sales, live events — that spikes far above steady state.
  • Disk I/O, CPU, or network on a single writer node is the stated bottleneck.

Solution ladder

  1. Vertical scaling + the right database. Bigger box first; consider write-optimized stores (LSM-tree databases like Cassandra) over B-tree RDBMSs.
  2. Shard and partition. Horizontal sharding splits data across instances by a partition key; vertical partitioning gives each entity type its own database. Choosing a partition key that spreads load evenly is the critical decision.
  3. Buffer bursts with queues; shed load. A queue (Kafka) absorbs spikes and lets the database drain at its own pace; under extreme load, explicitly reject or defer low-value writes.
  4. Batch and aggregate hierarchically. Combine many small writes into one (e.g. count events locally, flush every second; aggregate local → regional → global).

Tech to name & why

CassandraLSM storage engine is built for heavy sequential writes; masterless scaling.
KafkaDurable, high-throughput buffer between producers and the database.
DynamoDBManaged partitioning with per-partition throughput guarantees.
Redis countersIn-memory pre-aggregation before flushing batches downstream.

Deep dives to expect

  • Hot partitions — one key (a viral video) gets disproportionate writes; salt the key or split hot keys dynamically.
  • Resharding — adding shards later without downtime; consistent hashing limits data movement.
  • Queue trade-off — buffering adds latency and eventual consistency; say so before the interviewer does.

Ad click aggregatorYouTube Top-KRate limiterStravaMetrics/monitoringFB post search
Dealing with Contention▶ Many users compete for one resource — prevent double-booking without killing throughput.

Spot it when

  • Finite inventory: the last concert ticket, seat, or item in stock.
  • Auctions and bidding — the highest bid must win exactly once.
  • Any read-check-write sequence: two requests read "1 left", both validate, both decrement → oversold. The read and write aren't atomic.

Solution ladder

  1. Conditional (atomic) writes. Push the check into the write itself: UPDATE tickets SET qty = qty - 1 WHERE qty > 0. The database serializes it for you — solves most interview problems in one line.
  2. Pessimistic locking. SELECT … FOR UPDATE locks the row before modifying. Safe but holds locks — throughput suffers when everyone wants the same row.
  3. Optimistic concurrency control. Read a version number, write only if the version is unchanged, retry on conflict. Great when conflicts are rare; wasteful when they're constant.
  4. Distributed locks / queue serialization. When state spans multiple systems: Redis locks with TTLs, or funnel all operations on one resource through a queue so a single consumer applies them in order.

Prefer the single-database solution. Reaching for a distributed lock when one Postgres transaction would do is a classic overengineering flag.

Tech to name & why

Postgres transactionsACID + row locks handle most contention inside one database.
Redis (SETNX + TTL)Simple distributed lock; TTL prevents a crashed holder from deadlocking everyone.
Kafka (per-key ordering)Partition by resource ID → all ops on one item processed serially, no locks.
ZooKeeper/etcdBattle-tested coordination when you truly need distributed consensus.

Deep dives to expect

  • Deadlocks — with pessimistic locking, acquire locks in a consistent global order; use lock timeouts.
  • ABA problem — optimistic checks fooled by a value changing and changing back; use monotonic version numbers, not values.
  • Hot-resource throughput — everyone wants the same item; serialize via queue, or reserve with expiry (hold the ticket 10 minutes, release if unpaid).

Managing Long-Running Tasks▶ Work takes too long for a request — acknowledge instantly, process in the background.

Spot it when

  • Video transcoding, PDF/report generation, image resizing, CSV imports, bulk email.
  • Any operation that could exceed the typical 30–60 s web server timeout.
  • The user needs an immediate response but the work itself can wait.

The architecture — say it in one breath

  1. Accept & enqueue. The web server validates the request, pushes a job onto a queue, writes a pending row to a status table, and returns a job ID in milliseconds.
  2. Workers process. A separate pool of workers pulls jobs from the queue and does the heavy lifting — scaled independently of the web tier.
  3. Report status. Workers update the status database (pending → running → done/failed); the client polls the job ID or gets pushed an update (see Realtime Updates).

Tech to name & why

SQS / RabbitMQPurpose-built job queues: visibility timeouts, retries, dead-letter queues.
KafkaWhen jobs are events at huge scale and ordering/replay matter.
Redis (as queue)Lightweight option when you already run Redis; fine for modest scale.
Lambda / ECS workersElastic worker pools that scale with queue depth.

Deep dives to expect

  • Worker dies mid-job — visibility timeout returns the job to the queue; another worker retries.
  • Poison messages — after N failed retries, park the job in a dead-letter queue for inspection.
  • Duplicate execution — at-least-once delivery means workers must be idempotent (dedupe by job ID).
  • Backpressure — queue grows faster than workers drain it; autoscale workers, or shed/reject new jobs.
Don't overreach: if the work reliably finishes in < 1 s, synchronous is simpler and better. Introduce the queue only when timeouts or user experience demand it.

Seen in

YouTube uploads (transcoding)DropboxReport generationNewsletter sendLeetCode code judging
Multi-Step Processes▶ A workflow spans many services over hours or days — it must survive crashes mid-flight.

Spot it when

  • Order fulfillment: charge payment → reserve inventory → ship → confirm — any step can fail.
  • Flows with external dependencies (payment gateways) or human steps (approvals).
  • The interviewer asks "what if the server crashes between step 2 and step 3?"

Solution ladder

  1. Single-server orchestration. One service calls steps in order, persisting state in a database between steps. Fine until the orchestrator itself crashes or steps take days.
  2. Event-driven choreography. Each service reacts to the previous service's events (via Kafka). Decoupled, but the overall flow lives nowhere — debugging "where is order #123 stuck?" gets painful.
  3. Saga pattern. Each step has a compensating action (refund, release inventory) that runs to undo work when a later step fails — distributed "rollback" without distributed transactions.
  4. Workflow engine with durable execution. Define the flow declaratively; the engine checkpoints every step, resumes after crashes, and handles retries, timers, and versioning. The mature answer for anything complex.

Tech to name & why

TemporalDurable execution — workflows written as code, state checkpointed automatically, survives crashes and deploys.
AWS Step FunctionsManaged declarative state machines; native AWS service integration.
AirflowScheduled batch DAGs (data pipelines) — not per-user transactional flows.
Kafka (events)Backbone for choreography when loose coupling matters more than visibility.

Deep dives to expect

  • Crash mid-saga — state is persisted per step; the engine (or a recovery job) resumes or compensates.
  • Idempotency — retries must not double-charge; idempotency keys on every external call.
  • Versioning — deploying a new workflow definition while thousands are in flight.
  • External events — waiting days for a human approval or webhook; timers and timeouts.
Why interviewers love it: it separates system concerns (crashes, retries) from business concerns (what if the item is out of stock?) — mixing the two is what makes real systems brittle.

Pushing Realtime Updates▶ Clients need changes the moment they happen — the server must push, not wait to be asked.

Spot it when

  • Chat, live comments, collaborative editing, live dashboards, ride tracking.
  • The words "live", "instant", or "real-time" appear in the requirements.
  • Think in two hops: (1) how updates reach the client, (2) how updates reach the right server.

Hop 1 — server to client, simplest first

  1. Simple polling. Client asks every N seconds. Trivial to build, wasteful, latency = poll interval. A fine answer when "real-time" really means "within 30 seconds".
  2. Long polling. Server holds the request open until data arrives. Better latency, still HTTP-simple.
  3. Server-Sent Events (SSE). One long-lived HTTP connection, server streams one-way updates. The sweet spot for notifications, feeds, and progress bars — simpler than WebSockets.
  4. WebSockets. Persistent full-duplex connection. The answer when clients also send frequently: chat, games, collaborative editing. Costs: stateful servers, load-balancer support, reconnection logic.
  5. WebRTC. Peer-to-peer — reserve for audio/video calls and screen sharing.

Hop 2 — getting updates to the right server

  • Pub/Sub (Redis Pub/Sub, Kafka): every connection server subscribes to topics; publishers stay decoupled from wherever clients are connected. Default choice.
  • Consistent hashing: route each user/room to a specific stateful server so state lives with the connection — for heavier per-connection processing (e.g. a Google Doc's edit state).

Tech to name & why

WebSocketsFull-duplex; the only choice for frequent client→server traffic.
SSEOne-way push over plain HTTP; auto-reconnects; far less infra pain.
Redis Pub/SubFast fan-out of messages to whichever server holds the connection.
KafkaDurable pub/sub when updates must not be lost or need replay.

Deep dives to expect

  • Reconnection — clients drop and resume; need last-event IDs so missed messages replay.
  • Fan-out — a celebrity with millions of followers posts once; push to online users, lazily deliver to the rest.
  • Ordering — messages traversing different servers must render in sequence; per-conversation sequence numbers.
  • Connection state — millions of open sockets need heartbeats and a registry of who's connected where.

Handling Large Files▶ Uploads and downloads of big files — move bytes directly between client and storage.

Spot it when

  • Users upload videos, photos, or documents — anything over ~10 MB.
  • Routing file bytes through your API servers would make their bandwidth the bottleneck.
  • Remember: databases are great at structured data and queries, terrible at large binaries.

The architecture

  1. Presigned upload URL. Client asks your API "I want to upload X"; API writes a metadata row (status: uploading) and returns a short-lived presigned S3 URL scoped to that one object.
  2. Client uploads directly to S3. Bytes never touch your servers. For big files, use multipart upload: chunks upload independently and in parallel, failed chunks retry alone — this is what makes uploads resumable.
  3. Confirm completion. An S3 event notification (or client callback) flips the metadata row to ready.
  4. Download via CDN. Serve reads through CloudFront with presigned/signed URLs — edge latency, zero origin load.

Tech to name & why

S3 / GCS / Blob Storage11-nines durability, unlimited capacity, pay-per-object.
Presigned URLsTemporary, scoped credentials — direct transfer without exposing the bucket.
Multipart uploadParallel, resumable chunked uploads for GB-scale files.
CloudFront (CDN)Geographic edge caching for downloads.

Deep dives to expect

  • Failure at 99% — multipart means only the failed chunk retries; track uploaded chunks to resume.
  • Metadata sync — the database says a file exists but the upload never finished; reconcile with S3 events plus a sweeper for stale uploading rows.
  • Abuse — short URL expiry, content-length limits, per-user quotas, virus scanning post-upload.
  • Speed — parallel chunks on upload, CDN + range requests on download.

Proximity-Based Search▶ "Find things near me" — normal indexes can't answer two-dimensional questions.

Spot it when

  • Drivers near a rider, restaurants near a user, posts near a location.
  • Core insight to state: a B-tree sorts in one dimension, so an index on latitude gives you a horizontal strip of the whole world — 2D closeness can't be preserved in a 1D sort. That's why naive lat/long indexes force full scans.

Choose by data shape

  1. Small dataset? Skip the index. Under ~100K items, scan candidates and filter by distance. Only reach for geospatial indexing at real scale.
  2. Moving points (drivers, users) → encoded cells. Geohash converts lat/long into a sortable string, so a regular B-tree works; updates are cheap single-value writes. Query the cell plus its 8 neighbors (the "3×3 trick") to handle boundary edge cases.
  3. Shapes (zones, roads, polygons) → spatial trees. R-trees (PostGIS) index bounding rectangles and handle containment/intersection. Writes are pricier due to rebalancing — fine, because shapes rarely move.
  4. Always post-filter. The index only produces a candidate set; finish with exact distance math. Skipping this step is the classic mistake.

Tech to name & why

PostGISR-tree indexes on Postgres; the answer when geometry (polygons, zones) matters.
Redis GEOGeohash on sorted sets; fast candidate sets for live driver locations.
Elasticsearch geoBKD-tree geo fields; combine location with text/filter search.
Google S2Spherical cells, globally uniform size; powers MongoDB's 2dsphere.
Uber H3Hexagonal cells — six equidistant neighbors; great for rings and heat maps.

Deep dives to expect

  • Cell boundaries — two points meters apart can land in different geohash cells; that's what the 3×3 neighbor query fixes.
  • High-frequency updates — drivers move constantly; encoded cells win because an update is one integer write.
  • Precision picking — 5-char geohash ≈ 5 km, 9-char ≈ 5 m; match cell size to query radius.
Rule of thumb: spatial trees for shapes, encoded cells for moving points — and always, always post-filter by exact distance.

Likewise "Design Uber" = Proximity Search (matching) + Realtime Updates (tracking) + Multi-Step Processes (trip lifecycle), and "Design Ticketmaster" = Contention (seat booking) + Scaling Reads (event browsing) + Realtime Updates (seat maps). Name the patterns early — it shows the interviewer you have a map, not a memorized answer.

Design for failure — raise it before they ask

Distributed systems fail partially and constantly. Naming the failure modes and your recovery story unprompted is what separates a senior answer from a hopeful one.

Fanout — push vs pull decides where you pay the cost of delivering one event to many watchers.

Push · fan-out-on-writeprecompute per recipient at write time vs Pull · fan-out-on-readassemble the view at read time

Push gives fast reads, but a celebrity with 100M followers makes one write explode. Pull gives cheap writes, but every read redoes the work. Hybrid is the senior answer — push for normal accounts, pull for high-fanout ones, and merge at read time.

Robust & future-proof — the closing checklist

Before you wrap, sweep these to show the design survives growth to millions of users, not just today's load.

Handle millionsStateless tiers behind a load balancer, cached reads, sharded writes — no single node on the hot path.
Measure firstEstimate QPS, storage, and bandwidth up front; let the numbers choose the technology, not the reverse.
No SPOFEvery tier redundant across zones, with failover you've actually tested — not merely assumed.
Loosely coupledQueues and clear service boundaries so pieces scale, deploy, and fail independently.
ObservabilityMetrics, structured logs, tracing, and alerts — you can't scale what you can't see.
Design for changeVersioned APIs, backward-compatible migrations, and feature flags for safe, gradual rollout.
